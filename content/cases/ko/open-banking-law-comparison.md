## 요약

오픈뱅킹 규제는 표준화된 애플리케이션 프로그래밍 인터페이스(API)를 통해 전통적 은행과 제3자 제공업체(TPP) 간의 금융 데이터 공유를 의무화하거나 촉진하여, 금융 서비스의 경쟁 환경을 근본적으로 재편한다. 규제 접근법은 유럽연합의 결제서비스지침 2(PSD2, Directive (EU) 2015/2366) 및 그 후속인 PSD3(2023년 제안)에 따른 포괄적 입법 의무화에서부터, 미국 소비자금융보호국(CFPB)의 Dodd-Frank Act 제1033조를 이행하는 개인금융데이터권리규칙(Personal Financial Data Rights Rule)에 따른 발전하는 체계, 그리고 한국의 신용정보의 이용 및 보호에 관한 법률(Credit Information Use and Protection Act, 신용정보의 이용 및 보호에 관한 법률, 마이데이터 사업)에 따른 진보적 데이터 공유 의무까지 다양하다. 독일은 EU 회원국으로서 결제서비스감독법(Zahlungsdiensteaufsichtsgesetz, ZAG)을 통해 PSD2를 이행하며, 강력한 은행 부문이 준수 관행과 API 표준 개발 양면에서 영향을 미쳤다. 이러한 체계들은 소비자 동의, 데이터 이동성, API 표준화, 책임 배분 및 경쟁 정책이라는 상호 연결된 쟁점을 다루며, 금융 포용, 혁신 및 개인정보 보호에 중대한 함의를 갖는다.

## 한국

### 법률 체계

한국은 마이데이터(마이데이터) 사업을 통해 세계에서 가장 선진적인 오픈뱅킹 및 오픈파이낸스 체제 중 하나를 발전시켰으며, 데이터 이동성을 은행을 넘어 보험, 증권, 통신 및 정부 서비스까지 확장하였다:

- **신용정보의 이용 및 보호에 관한 법률(신용정보의 이용 및 보호에 관한 법률, 2020년 개정):** 한국 마이데이터 체제의 기반 법률로, 2020년에 오픈파이낸스를 가능하게 하기 위해 포괄적으로 개정되었다:
  - **제2조 -- 정의:** "개인신용정보"를 금융 거래 기록, 신용평가, 보험 및 증권 데이터, 기타 금융 관련 개인 데이터를 포괄하도록 광범위하게 정의한다.
  - **제22조의9 -- 마이데이터 사업:** 마이데이터 사업 허가 범주(개인신용정보 전송요구권)를 설립하여, 허가받은 사업자가 복수 금융기관으로부터 소비자 동의에 기반한 데이터를 수집, 분석하고 맞춤형 금융 서비스를 제공할 수 있도록 한다.
  - **제33조의2 -- 데이터 이동권(전송요구권):** 개인에게 데이터 보유자(은행, 보험사, 증권사)에 대해 자신의 개인신용정보를 표준화된 전자 형식으로 허가받은 마이데이터 사업자 또는 본인에게 전송하도록 요청할 권리를 부여한다.
  - **제39조의3 -- API 표준:** 금융위원회에 데이터 전송을 위한 API 표준 및 기술 사양을 규정할 권한을 부여하여, 금융기관 간 상호운용성을 보장한다.
- **전자금융거래법(전자금융거래법):** 전자 결제 서비스를 규율하고 결제 개시 및 계좌 정보 서비스의 법적 체계를 수립하여 마이데이터 체제를 보완한다.
- **금융혁신지원 특별법(금융혁신지원 특별법, 2019):** 혁신적 금융 서비스를 위한 규제 샌드박스를 설립하여, 핀테크 기업이 최대 4년간 완화된 규제 조건 하에서 오픈뱅킹 애플리케이션을 시험할 수 있도록 한다.

### 주요 원칙

한국의 마이데이터 시스템은 2021년 12월 1일 공식 출범하였으며, 금융위원회가 마이데이터 사업자의 허가 및 감독을 담당한다. 2025년 현재 전통적 은행, 핀테크 스타트업 및 기술 기업을 포함하여 60개 이상의 사업자가 마이데이터 사업 허가를 보유하고 있다. 이 시스템은 한국금융결제원(KFTC, 금융결제원)이 운영하는 중앙집중식 API 인프라를 사용하여 데이터 교환을 위한 표준화된 인터페이스를 제공한다.

마이데이터 체제는 유럽 또는 미국의 동등한 체제보다 데이터 범위가 현저히 넓다:

- **1단계(2021-2022):** 은행 거래 데이터, 신용카드 데이터 및 보험 데이터.
- **2단계(2023-2024):** 증권 데이터, 통신 데이터, 공공요금 데이터.
- **3단계(2025-2026):** 의료 데이터, 정부 행정 데이터로 확장하여 포괄적 개인 데이터 이동성 생태계를 구축.

금융보안원(FSI, 금융보안원)은 마이데이터 API 표준을 발행 및 관리하며, OAuth 2.0 인증을 갖춘 JSON 기반 RESTful API를 의무화하고, 모든 적용 대상 금융기관에 전용 API 엔드포인트 구현을 요구한다.

### 주요 사례

- **금융위원회 마이데이터 허가 취소 -- 뱅크샐러드(2024):** 금융위원회는 취약성 평가에서 수집된 소비자 금융 데이터가 동의한 보존 기간을 초과하여 저장되고 있음이 밝혀져 신용정보법 제20조의2를 위반한 데 대해, 주요 마이데이터 사업자 뱅크샐러드(뱅크샐러드)에 부적절한 데이터 보안 관행으로 인한 경고 조치를 내렸다.
- **토스 데이터 이동성 확장 사건(2023):** 토스(비바리퍼블리카)는 마이데이터 체계에서 전송 가능한 데이터 범위를 보험증권 세부사항 및 증권 포트폴리오 데이터로 확장하도록 금융위원회에 성공적으로 청원하여 2단계 출시를 가속화하였다.
- **KB국민은행 API 준수 명령(2023):** 금융감독원은 KB국민은행의 전용 인터페이스가 마이데이터 API 기술 사양에서 요구하는 99.5% 가용성 기준을 지속적으로 충족하지 못한 후, 전자금융거래법에 따른 시정 명령을 부과하여 API 성능 결함을 시정하도록 명령하였다.

## 미국

### 연방 체계

미국은 CFPB의 Dodd-Frank Act 제1033조 이행을 통해 의무적 오픈뱅킹 체계를 향해 나아가고 있다:

- **Dodd-Frank 월가 개혁 및 소비자보호법, 제1033조(12 U.S.C. § 5533):** 2010년에 제정된 제1033조는 적용 대상자(은행, 신용조합, 비은행 금융회사)가 소비자 요청 시 해당 적용 대상자로부터 취득한 소비자 금융상품 또는 서비스에 관한 정보를 제공해야 한다고 규정하며, 이에는 거래 데이터, 계좌 잔액 정보 및 계좌에서 결제를 개시하는 데 필요한 정보가 포함된다.
- **CFPB 개인금융데이터권리규칙(최종 규칙, 2024년 10월):** CFPB는 오랫동안 기다려온 제1033조 이행 규정을 확정하여 다음을 수립하였다:
  - **적용 데이터:** 거래 정보, 계좌 잔액 데이터, 결제 개시 정보, 소비자 금융상품에 적용되는 조건. 기밀 상업 정보, 독점 알고리즘, 사기 방지를 위해 수집된 정보는 제외.
  - **적용 데이터 제공자:** 예금취급기관, 비예금취급 금융기관, 신용카드 발급사 및 소비자 금융 데이터를 통제하는 기타 사업체. 자산 규모에 따라 단계적 이행.
  - **제3자 접근:** 승인된 제3자는 소비자의 명시적이고 충분한 정보에 기반한 동의에 따라 개발자 인터페이스(API)를 통해 소비자 금융 데이터에 접근할 수 있다. 이 규칙은 제3자가 승인된 목적에 "합리적으로 필요한" 범위로 데이터 수집을 제한하도록 요구한다.
  - **소비자 승인 및 철회:** 소비자는 데이터 접근을 적극적으로 승인해야 하며 언제든지 승인을 철회할 수 있다. 승인은 매년 갱신되어야 하며, 재승인 시 소비자의 적극적 행위가 요구된다.
  - **데이터 보안:** 적용 데이터 제공자와 승인된 제3자는 Gramm-Leach-Bliley Act(15 U.S.C. § 6801) 안전장치를 포함한 관련 법률에 부합하는 보안 관행을 유지해야 한다.
  - **이행 일정:** 최대 데이터 제공자(자산 5,000억 달러 초과)는 2026년 4월까지 준수해야 하며; 중형 기관은 2027년까지; 소형 기관은 2028-2030년까지.
- **Gramm-Leach-Bliley Act(GLBA, 15 U.S.C. §§ 6801-6809):** 오픈뱅킹 데이터 공유와 상호작용하는 금융 개인정보 보호 및 데이터 보안 요건을 수립하여, 금융기관이 행정적, 기술적, 물리적 안전장치를 통해 소비자 금융 정보를 보호하도록 요구한다.

### 업계 표준

단일 의무적 API 표준이 없는 상황에서, 미국 오픈뱅킹 생태계는 업계 주도의 표준화에 의존해 왔다:

- **Financial Data Exchange(FDX):** 70개 이상의 금융기관 및 데이터 수집자가 채택한 FDX API 표준을 개발한 비영리 업계 컨소시엄. FDX 6.0(2024)은 CFPB 제1033조의 완전한 준수를 지원한다.
- **스크린 스크래핑 단계적 폐지:** CFPB 규칙은 사실상 스크린 스크래핑(인증서 기반 데이터 수집)에서 API 기반 접근으로의 전환을 의무화하여, 오래된 보안 및 신뢰성 우려를 해결한다. 데이터 제공자는 상업적으로 합리적인 성능 기준을 갖춘 개발자 인터페이스를 유지해야 한다.

### 주요 사례

- **CFPB v. TransUnion (2024):** CFPB는 TransUnion이 공정신용보고법(FCRA)에 따라 소비자에게 정확한 금융 데이터 접근을 제공하지 못했다고 주장하며 집행 조치를 취하였으며, 이는 제1033조 체계에 따른 데이터 이동성 의무에 함의를 갖는다.
- **Plaid, Inc. 소송(2021-2023):** 다수의 집단소송에서 미국의 주요 데이터 수집 중개업체인 Plaid가 은행 로그인 화면을 모방한 기만적 인터페이스를 통해 소비자 뱅킹 인증서를 수집하여 소비자가 승인한 것보다 더 많은 데이터에 접근하였다고 주장하였다. Plaid는 2022년 통합 집단소송에서 5,800만 달러에 합의하였다.
- **Capital One v. 데이터 수집자 분쟁(2020-2023):** Capital One의 데이터 수집자에 의한 스크린 스크래핑 접근 차단과 이후 API 기반 접근에 대한 협상은 은행의 데이터 통제와 소비자 데이터 이동성 권리 간의 긴장을 부각시켰으며, CFPB의 규칙 제정 과정에 직접적으로 기여하였다.
- **소비자금융데이터권리 규칙제정 소송(2025):** 은행 업계 단체(미국은행협회, Bank Policy Institute)는 CFPB의 제1033조 규칙이 CFPB의 법적 권한을 초과하고 불합리한 준수 부담을 부과한다고 주장하며 법적 이의를 제기하였으며, 특히 데이터 접근을 무료로 제공해야 하는 의무에 관해 문제를 제기하였다.

## 독일

### 법률 체계

독일은 국내 법률을 통해 EU의 오픈뱅킹 체계를 이행하며, 상업은행, 저축은행(Sparkassen) 및 협동조합은행(Genossenschaftsbanken)의 3축 체계를 포함한 은행 부문의 특성이 이행 역학에 영향을 미친다:

- **결제서비스감독법(Zahlungsdiensteaufsichtsgesetz, ZAG):** PSD2를 독일법으로 전환하여, 다음을 포함한 결제 서비스의 규제 체계를 수립한다:
  - **ZAG 제1조 제1항 -- 결제 서비스:** 계좌정보서비스(Kontoinformationsdienste)와 결제개시서비스(Zahlungsausloesedienste) -- PSD2에 의해 신설된 두 가지 서비스 유형 -- 를 포함한 규제 대상 결제 서비스를 정의한다.
  - **ZAG 제48조 -- 결제 계좌 접근(XS2A):** PSD2 제66조(결제 개시)와 제67조(계좌 정보)를 이행하여, 계좌관리 결제서비스제공자(ASPSP)가 전용 인터페이스를 통해 또는, 그것이 불가능한 경우 고객 대면 인터페이스(대체 메커니즘)를 통해 승인된 제3자 제공업체에 접근을 허가하도록 요구한다.
  - **ZAG 제49조 -- 강력한 고객 인증(SCA):** PSD2의 SCA 요건을 이행하여, 전자 결제 거래 및 계좌 접근에 다요소 인증을 요구한다.
- **독일 은행법(KWG):** BaFin은 KWG의 허가 체계에 따라 오픈뱅킹 서비스를 제공하는 결제 기관 및 핀테크 기업을 감독한다.
- **BGB 제675c조~제676c조(결제 서비스 조항):** 무단 거래에 대한 책임 배분(제675u조: PSP 책임, 소비자는 중과실이 없는 한 최대 EUR 50 부담)과 정보 의무를 포함한 결제 서비스 계약의 민법 체계.
- **GDPR 및 BDSG:** 오픈뱅킹에 따른 데이터 공유는 적법한 처리 근거(제6조), 목적 제한(제5조 제1항 (b)), 정보주체의 권리(제15-22조)를 포함한 GDPR 요건을 준수해야 한다. PSD2 동의와 GDPR 동의의 교차는 상당한 준수 과제가 되고 있다.

### 주요 원칙

독일의 PSD2 이행은 몇 가지 특징적 요인에 의해 형성되었다:

1. **Berlin Group NextGenPSD2 표준:** 독일은 Berlin Group의 NextGenPSD2 API 체계를 주도해 왔으며, 이는 특히 독일신용산업위원회(Deutsche Kreditwirtschaft)가 공동 운영하는 인터페이스를 통해 독일 은행에서 널리 채택되었다.
2. **강력한 고객 인증(SCA) 이행:** 독일 은행들은 앱 기반 인증(pushTAN)과 생체인식 검증을 결합한 방식을 주요 접근법으로 하여 SCA 솔루션에 대규모 투자를 하였다. 독일연방은행은 SCA 이행으로 2020년에서 2024년 사이 독일의 결제 사기 손실이 35% 감소했다고 보고하였다.
3. **저축은행 및 협동조합은행 준수:** 900개 이상의 개별 기관으로 구성된 독일 저축은행 및 협동조합은행 부문의 분산 구조는 독특한 이행 과제를 제시하였다. 중앙집중식 IT 서비스 제공업체(저축은행을 위한 Finanz Informatik; 협동조합은행을 위한 Atruvia)가 규모의 경제를 달성하며 공유 API 인프라를 개발하였다.

### 주요 사례

- **BaFin N26 오픈뱅킹 준수 명령(2022):** BaFin은 디지털 은행 N26에 제3자 제공업체를 위한 전용 인터페이스의 결함을 시정하도록 명령하였으며, 동 은행의 API가 불충분한 가용성(99% 목표 미달)과 승인된 계좌정보서비스 제공업체에 대한 접근을 사실상 거부하는 불합리한 요율 제한을 부과하여 ZAG 제48조를 위반한 것으로 판단하였다.
- **연방대법원(BGH) 스크린 스크래핑 판결(2020, XI ZR 535/18):** BGH는 PSD2에 따른 스크린 스크래핑 전환 기간에 대해 판결하여, Commission Delegated Regulation (EU) 2018/389의 기술 요건을 충족하는 전용 인터페이스가 완전히 운영되기 전에 은행이 승인된 결제개시서비스 제공업체에 의한 스크린 스크래핑 접근을 일방적으로 차단할 수 없다고 판시하였다.
- **독일연방은행 PSD2 면제 판결(2023):** 독일연방은행은 PSD2 SCA 면제의 관할 기관으로서 Delegated Regulation 제18조에 따른 거래 위험 분석(TRA) 면제의 적용을 명확히 하는 결정을 발표하여, 결제서비스제공자의 사기율이 규정된 임계값 이하로 유지되는 경우 EUR 500 미만의 저위험 거래가 SCA 없이 진행될 수 있도록 하였다.
- **BaFin 핀테크 허가 절차(2024):** BaFin은 ZAG에 따라 API 기반 금융 데이터 중개 서비스를 제공하고자 하는 여러 독일 핀테크 기업에 결제기관 허가를 부여하여, 오픈뱅킹 서비스 제공업체에 대한 규제 취급의 선례를 수립하였다.

## ECHR / EU 체계

### 협약 조항

유럽인권협약은 오픈뱅킹 규제에 적용 가능한 기본 원칙을 제공한다:

- **제8조 -- 사생활 및 가정생활 존중 권리:** 금융 데이터 공유는 사생활의 구성요소로서 개인 데이터에 대한 제8조의 보호와 관련된다. ECtHR은 금융 정보가 제8조에 의해 보호되는 개인 데이터에 해당함을 인정하였다(M.N. and Others v. San Marino, 2015, App. No. 28005/12). 모든 의무적 데이터 공유 체제는 제8조 제2항의 비례성 요건을 충족해야 한다.
- **제1의정서 제1조 -- 재산 보호:** 은행이 보유한 금융 데이터는 은행(독점적 사업 정보로서)과 소비자(권리를 가진 개인 데이터로서) 양측의 "재산"에 해당할 수 있다. 오픈뱅킹 의무를 통한 해당 데이터의 강제적 공유는 제1의정서 보호를 관여시키며, 데이터 공유가 정당한 목적에 기여하고 비례성을 유지하도록 요구한다.
- **제6조 -- 공정한 재판을 받을 권리:** 오픈뱅킹 체계에 따른 규제 집행의 대상이 되는 소비자 및 금융기관은 규제 결정에 대한 사법 심사 접근을 포함한 공정한 절차를 받을 권리를 갖는다.

### EU 입법 체계

- **PSD2(Directive (EU) 2015/2366):** EU 오픈뱅킹의 기반 법률인 PSD2는 두 가지 핵심 혁신을 도입하였다: (1) 허가받은 제3자 제공업체(TPP)가 소비자 동의에 따라 계좌정보서비스(AIS)를 통해 소비자 결제 계좌 데이터에 접근하고 결제개시서비스(PIS)를 통해 결제를 개시할 수 있는 권리; (2) 전자 결제에 대한 강력한 고객 인증(SCA) 요건. PSD2는 2019년 9월 14일부터 모든 EU 회원국에서 완전히 시행되었으며, SCA 조항은 2020년 12월 31일부터 적용되었다(연장된 마이그레이션 기간 이후).
- **PSD3 및 결제서비스규정(PSR)(2023년 6월 제안):** 유럽위원회는 PSD2의 인식된 결함을 해결하기 위해 개정 지침(PSD3)과 직접 적용 가능한 규정(PSR)으로 PSD2를 대체할 것을 제안하였다:
  - **강화된 API 성능 기준:** PSR은 전용 인터페이스에 대한 양적 성능 기준을 의무화하여, 은행의 부진한 API 성능을 통한 TPP 접근의 "소프트 차단"을 해결한다.
  - **IBAN/성명 검증:** 신용 이체에 대한 수취인 일치 확인을 의무화하여 사기를 줄인다.
  - **오픈파이낸스 확장:** PSD3/PSR이 결제 계좌에 초점을 맞추는 반면, 제안된 금융정보데이터접근(FIDA) 규정은 데이터 공유를 보험, 연금, 투자 및 신용 데이터로 확장하여 오픈뱅킹에서 오픈파이낸스로 전환한다.
  - **책임 체계 개혁:** PSR은 무단 거래에 대한 ASPSP와 TPP 간의 책임 배분을 명확히 하여, 기관 간 분쟁을 야기한 PSD2의 모호성을 해결한다.
- **금융정보데이터접근(FIDA) 규정(2023년 6월 제안):** FIDA 제안은 오픈뱅킹 모델을 포괄적 오픈파이낸스 체계로 확장한다:
  - 금융 데이터 보유자(은행, 보험사, 투자회사, 연금 제공자)가 표준화된 인터페이스를 통해 승인된 금융정보서비스 제공업체(FISP)에 소비자 금융 데이터를 제공하도록 요구한다.
  - 기술 표준 개발, 분쟁 해결 및 보상 체계 설정을 위한 업계 주도 거버넌스 기구로서 "금융 데이터 공유 계획"을 수립한다.
  - 금융 데이터 공유를 위한 부문별 규칙을 수립하면서 GDPR 데이터 이동성 원칙과 정합성을 유지한다.
- **GDPR(Regulation (EU) 2016/679):** GDPR의 데이터 보호 체계는 오픈뱅킹과 광범위하게 교차한다:
  - **제6조 제1항 (c) -- 법적 의무:** 금융기관은 PSD2/PSR 요건에 따른 데이터 공유의 적법한 근거로 법적 의무에 의존할 수 있다.
  - **제20조 -- 데이터 이동권:** PSD2의 데이터 접근 조항을 보완하나, GDPR 데이터 이동성과 PSD2 접근권의 범위가 완벽하게 일치하지는 않는다.
  - **제7조 -- 동의 조건:** 동의가 데이터 공유의 적법한 근거인 경우, 자유롭게 주어진, 구체적이고, 충분한 정보에 기반한, 명확한 동의에 대한 GDPR 요건이 충족되어야 한다.

### 주요 판례법 및 결정

- **EBA의 PSD2 API 성능에 대한 의견(2023):** 유럽은행감독청은 EU 전역의 전용 인터페이스 중 30%가 허용 가능한 성능 기준을 충족하지 못한다는 의견을 발표하여, 각국 관할 기관이 미준수 계좌관리 PSP에 대해 집행 조치를 취할 것을 권고하고 PSR에 양적 API 성능 기준의 포함을 지지하였다.
- **CJEU 사건 C-306/20, Heureka Group (2023):** 직접적인 오픈뱅킹 사건은 아니지만, 경쟁법에 따른 플랫폼 데이터 접근에 관한 CJEU의 판결은 시장지배적 금융기관이 공정하고 합리적이며 비차별적인 조건으로 데이터 접근을 제공해야 하는 의무와 관련된 원칙을 확립하였다.
- **EDPB의 PSD2와 GDPR 상호작용에 관한 지침(2020, Guidelines 06/2020):** EDPB는 PSD2의 동의 요건과 GDPR의 데이터 보호 원칙 간의 상호작용을 명확히 하는 지침을 채택하여, 계좌 접근에 대한 PSD2 동의가 자동으로 GDPR 동의 요건을 충족하지 않으며, 데이터 최소화 원칙이 TPP 데이터 수집에 적용됨을 확립하였다.
